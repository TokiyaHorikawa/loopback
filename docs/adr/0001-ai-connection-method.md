# ADR-0001: AI接続方式としてMCP (SSE) を採用する

- **ステータス**: 承認
- **日付**: 2026-03-14

---

## コンテキスト

LoopbackはローカルでDockerコンテナとして動作する。Claude Code等のAIエージェントがLoopbackのデータにアクセスし、ふりかえりに伴走できるようにしたい。

接続方式を決める必要がある。

## 検討した選択肢

### 1. ローカルHTTP API (REST)

DockerコンテナがREST APIを公開し、Claude Codeがツールとして呼び出す。

- (+) 汎用的で理解しやすい
- (-) Claude Code側でツール定義を自前で記述する必要がある
- (-) APIスキーマの設計・ドキュメント整備が必要

### 2. MCP (SSE)

DockerコンテナがMCPサーバー（SSE）をローカルに公開し、Claude Codeが接続する。

- (+) Claude Codeが既にMCPクライアントとして動作するため、設定ファイルにURLを追加するだけで接続可能
- (+) Dockerコンテナとの相性が良い（ポート公開のみ）
- (+) ツール定義がMCPプロトコルに標準化されており、自前で設計する必要がない
- (+) Web UIのバックエンドとMCPサーバーを同一アプリ内で共存させやすい

### 3. CLI (`docker exec`)

Dockerコンテナ内のCLIコマンドを`docker exec`経由で呼び出す。

- (+) 実装がシンプル
- (-) `docker exec`を挟む分、取り回しが悪い
- (-) Claude Codeとの統合に余計な設定が必要

## 決定

**AI接続方式としてMCP (SSE) を採用する。**

ローカルで動くDockerコンテナがMCP (SSE) エンドポイントをローカルに公開し、Claude Code等のAIエージェントがそこに接続する。ユーザーはClaude Codeの設定ファイルにエンドポイントのURLを追加するだけで利用できる。

なお、Web UIのバックエンドはREST APIとして実装し、MCPサーバーはその上のレイヤーとして共存させる。

## 根拠

- **Claude Codeとの統合コストが最小** — 設定ファイルへのURL追加だけで接続できる。ツール定義をユーザーが自前で書く必要がない
- **MCPプロトコルによる標準化** — ツールの定義・呼び出し規約がMCPに標準化されており、設計コストが低い
- **Dockerと相性が良い** — コンテナのポートを公開するだけで実現できる。`loopback start` で全コンポーネントと同時に起動できる
- **Web UIとの共存が容易** — REST APIをベースにして、MCPレイヤーを追加する構成が自然
- **AIエコシステムとの親和性** — MCP対応のAIエージェントであれば、Claude Code以外でも接続できる拡張性がある
