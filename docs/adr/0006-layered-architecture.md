# ADR-0006: サーバーサイドをレイヤードアーキテクチャに分割する

- **ステータス**: 承認
- **日付**: 2026-03-15

---

## コンテキスト

ADR-0004 で決定した Hono + SQLite の構成で API を実装してきた。初期実装ではルートハンドラにバリデーション・ビジネスロジック・DBアクセスをすべて記述していた。コードが増えるにつれ、1ファイルに複数の関心が混在し、変更時の影響範囲が読みにくくなってきた。

classベースの fat model（Rails way）も検討したが、本プロジェクトは関数ベースで書いており、classに寄せる理由がない。関数ベースで単一責任の原則（SRP）を適用するために、ファイル＝責務の単位としてレイヤー分割を行う。

## 検討した選択肢

### 1. Fat Model（エンティティ単位で1ファイルにまとめる）

```
models/goals.ts    → validate + query + insert すべて
models/reviews.ts  → validate + query + insert すべて
```

- (+) ファイル数が少ない
- (-) 1ファイルに複数の関心が混在する（バリデーション、クエリ構築、ビジネスルール）
- (-) 関数が増えるとファイルが肥大化し、関数ベースでは凝集の境界が曖昧になる

### 2. Vertical Slices（ユースケース単位で分割）

```
features/goals/list-goals.ts
features/goals/create-goal.ts
```

- (+) 1ファイル = 1ユースケースで変更範囲が明確
- (-) 共通クエリ（`getGoalIds` など）の置き場所に悩む
- (-) このプロジェクトの規模では細かすぎる

### 3. レイヤードアーキテクチャ（責務ごとに分割）

```
routes/       → HTTP の関心（リクエスト/レスポンス）
validators/   → 入力バリデーション
services/     → ビジネスロジック・ユースケース
repositories/ → DBアクセス（クエリ構築・永続化）
```

- (+) 各ファイルが単一の関心を持ち、SRPが自然に適用される
- (+) テスト時にレイヤーを差し替えやすい
- (+) 変更理由が明確（「バリデーションルールの変更」→ validators のみ、「クエリの最適化」→ repositories のみ）
- (-) 小規模プロジェクトではボイラープレートが増える

## 決定

**選択肢 3: レイヤードアーキテクチャを採用する。**

関数ベースのコードにおいて、ファイル＝凝集の単位と定義し、レイヤーごとにファイルを分割することで SRP を実現する。

---

## レイヤー定義

### routes（HTTPレイヤー）

- リクエストの受け取りとレスポンスの返却のみを行う
- バリデーション関数を呼び、serviceを呼び、結果をJSONで返す
- HTTPステータスコードの判断はここで行う

### validators（入力バリデーション）

- リクエストボディの検証とパースを行う
- 型安全な入力オブジェクトを返す
- DBには一切アクセスしない

### services（ビジネスロジック）

- ユースケースのオーケストレーションを行う
- 複数のrepositoryを組み合わせた処理（例: review作成 + review_goals の紐づけ）はここで行う
- 単純なCRUDの場合はrepositoryをそのまま呼ぶだけでよい（無理にロジックを足さない）

### repositories（データアクセス）

- DBへのクエリ構築と実行のみを行う
- drizzle の操作はすべてここに閉じ込める
- ビジネスルールの判断はしない

## ディレクトリ構造

```
packages/server/src/
  routes/
    goals.ts
    reviews.ts
    context.ts
  validators/
    goals.ts
    reviews.ts
  services/
    goals.ts
    reviews.ts
  repositories/
    goals.ts
    reviews.ts
```

---

## 依存の方向

```
routes → validators
routes → services → repositories
```

- 上位レイヤーは下位レイヤーに依存してよい
- 下位レイヤーは上位レイヤーに依存してはならない
- 同一レイヤー間の依存は許容する（例: `services/reviews.ts` が `repositories/goals.ts` を使う）

---

## 根拠

- **変更容易性** — バリデーションルールの変更、クエリの最適化、ビジネスロジックの追加がそれぞれ独立したファイルに閉じる。変更理由が1つのファイルに対して1つになる
- **関数ベースでのSRP** — classではメソッドの凝集をclassが担保するが、関数ベースではファイルがその役割を担う。レイヤー分割によりファイルの責務を明確にする
- **テスタビリティ** — repositoryを差し替えればserviceを単体テストできる。バリデーションはHTTPなしでテストできる
- **学習効果** — レイヤードアーキテクチャはソフトウェア設計の基本パターンであり、理解しておく価値がある
